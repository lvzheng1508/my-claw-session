# 规格驱动开发(SDD):"设计先行"理念的演进与反思


## 引言

当 AI 编码助手能在几秒内生成一个完整的功能模块时，一个问题变得前所未有的紧迫：**谁来保证 AI 写的代码做的是对的事？**

这不是一个新问题。从 1960 年代软件工程诞生之初，"先定义、再实现"就是一种反复出现的工程倾向--**设计先行**。六十年间，它并不是沿着一条笔直路线不断升级，而是在不同背景下以不同实践形态反复出现：有时是数学化的形式化规格，有时是代码级的契约与测试，有时是接口契约和服务边界，而今天则出现了 AI 时代的可执行规格。形式各异，内核相通。

而**规格驱动开发(Specification-Driven Development, SDD)**，可以被视为这种"设计先行"倾向在 AI 时代的一种新实践。它继承了"设计先行"的精神，但借助 AI 的能力，让自然语言规格的编写、补全和维护第一次在更大范围内变得可实用。

SDD 的核心理念并不复杂——在动手实现之前，先用精确的语言定义"系统应该做什么"。但围绕这个核心理念，过去六十年诞生了数十种方法论、数百种工具、无数种实践路径。它们各有取舍，各有成败，各有适用的边界。

本文试图做一件事：**站在 2026 年的时间节点，回看过去几十年中几种具有代表性的"设计先行"实践，理解 SDD 与它们之间的相似性与差异，并回答一个对当下每一个技术团队都至关重要的问题——在 AI 编码盛行的今天，我们该如何看待 SDD?**


## 一、设计先行:从形式化方法到规格驱动

软件工程中一直反复出现一种持久的工程倾向：**设计先行**--在写代码之前，先以某种形式定义好系统应该做什么、怎么做。六十年来，这种倾向从未中断，只是在不同阶段借由不同方法被重新表达。

从 1960 年代的数学公式，到 1990 年代的代码契约，到 2000 年代的测试用例，到 2010 年代的 API 接口，再到今天 AI 时代的自然语言规格——这些实践未必构成一条严格线性的演化链，但它们共享一个稳定内核：**先定义，再实现。**

### 1.1 早期年代(1960s-1980s):用数学定义正确性

软件危机催生了软件工程，也催生了"设计先行"最早的形式——"形式化规格"(Formal Specification)。

**Z Notation**(1970s)基于集合论，用数学符号精确描述系统的状态和操作 [1]。**VDM**(Vienna Development Method)由 IBM 维也纳实验室提出，是最早的系统化形式化方法之一 [2]。**TLA+**(Leslie Lamport)用数学逻辑描述并发系统的行为 [3]。**B-Method**(Jean-Raymond Abrial)更进一步，将形式化规格与逐步细化结合，能够在数学上证明实现是正确的 [4]。

这一时期的核心信念是：**如果规格足够精确，就能在写代码之前证明系统是正确的。**

这个信念在理论上成立，在实践中却碰了壁。形式化方法的门槛极高，开发者需要先学数学建模;像性能、可用性这类"非功能性需求"很难用数学表达;而且跟当时的开发流程严重脱节。最终，形式化方法退守到了"不能死机"的领域——航空航天、铁路信号、核能控制、芯片验证。

但它们从未消失。Amazon 用 TLA+ 验证 DynamoDB 的分布式设计，发现了手工评审无法发现的深层并发 bug [5]。Intel 用形式化方法验证芯片设计。Microsoft 用 TLA+ 验证 Azure 基础设施。这些实践证明：**形式化规格不是银弹，但在某些高风险场景下，它仍然具有难以替代的价值。**

只是，对于绝大多数软件项目来说，它太重了。"设计先行"需要找到一条更亲民的路。

### 1.2 黄金时代(1990s-2000s):从数学到代码，从精英到大众

1990 年代，"设计先行"开始寻找更亲民的形态。核心思路是：**既然数学语言太重，那就用代码语言来表达设计意图。**

**Design by Contract**(Bertrand Meyer, 1986 提出，1990s 在 Eiffel 语言中推广)是一个里程碑 [6]。它的核心思想是：把规格直接写进代码里——用"前置条件"(调用前必须满足什么)、"后置条件"(执行后必须保证什么)和"不变式"(任何时候都不能违反什么)来定义函数的契约。调用者必须满足前置条件，实现者必须保证后置条件。规格不再是独立于代码的文档，而是代码的一部分。

这一思想深刻影响了后世。Java 的 assert、Python 的 type hints、Rust 的类型系统——它们在不同程度上都是 Design by Contract 的继承者。Rust 的类型系统甚至被描述为"编译期的契约"。

2000 年代初，**Test-Driven Development**(Kent Beck, 2002)成为"设计先行"的主流实践 [7]。TDD 的循环——先写一个会失败的测试(Red)，再写最少的代码让测试通过(Green)，最后重构代码(Refactor)--本质上就是把"系统应该做什么"变成了可以自动验证的测试。测试用例就是设计意图的编码表达，测试通过就是设计被满足。

TDD 的伟大在于它**大幅降低了"设计先行"的门槛**。你不需要懂数学，不需要学形式化语言，只需要会写代码就能定义设计意图。Google 在内部大规模推广 TDD,ThoughtWorks 将其作为核心工程实践，Microsoft 推荐内部团队采用。

同年，**Behavior-Driven Development**(Dan North, 2003)进一步降低了门槛 [8]。BDD 用"Given(前提)-When(操作)-Then(预期结果)"的自然语言格式描述行为，让非技术人员也能参与规格定义。Cucumber [9]、Behave、SpecFlow 等工具让 BDD 场景可以自动转化为可执行测试。LinkedIn 在大规模 BDD 实践中构建了完整的自动化验收测试体系。

**Specification by Example**(Gojko Adzic, 2011)则从另一个角度切入 [10]:不写抽象规格，而是用**具体的业务实例**来定义系统行为。eBay、Guardian 等企业用 SbE 将业务、开发、测试三方拉到同一张桌前，用例子代替争论。

### 1.3 云原生时代(2010s):接口即契约

2010 年代，微服务的流行让"设计先行"的重心从"系统内部行为"转移到了"服务间通信边界"。

**API-First Development** 成为微服务架构的事实标准 [11]。Stripe、Twilio、Google 等公司将 API 契约视为产品本身——先定义 OpenAPI/GraphQL Schema，再生成文档、SDK、测试。Stripe 的 API 设计甚至成为业界标杆，其版本控制和兼容性策略被广泛模仿。

**消费者驱动的契约测试**让规格变成了服务间的一份"合同"。Pact(最初在澳洲 REA Group 诞生)[12] 的做法是：由调用方定义"我期望你给我什么"，被调用方自动验证自己是否满足这个期望。Netflix 在其微服务架构中大规模采用契约测试，实现了服务的独立部署。

**Protocol Buffers**(Google)[13] 和 **Thrift**(Facebook/Meta)[14] 成为服务间通信的契约语言。Google 将 Protocol Buffers 作为全公司的 RPC 契约标准，Meta 用 Thrift 连接数千个内部服务。

这个时代的关键特征是：**"设计先行"不再是开发者的个人习惯，而是组织级别的基础设施。** 当系统被拆分成几十上百个微服务时，没有契约规格，集成就变成了一场灾难。

### 1.4 AI 时代(2024+):规格驱动开发(SDD)

为什么 SDD 在今天突然"火"了？因为它解决了一个 AI 编码时代最迫切的问题。

过去，"设计先行"长期处于"理论很美好，现实很骨感"的尴尬境地——人工编写和维护设计文档的成本极高，文档写完了代码还没开始写，代码一改文档就废了(规格腐烂)。很多团队不是不想做，而是成本太高，难以持续。

AI 改变了这个成本结构。

2024-2025 年，AI 编码助手的爆发式增长催生了两个变化：

当 GitHub Copilot、Cursor、Claude Code、Codex 等工具能快速生成代码时，一个老问题以新面貌出现：**AI 理解你想要什么吗？**

"Vibe coding"——凭感觉让 AI 写代码——在原型探索中很有价值，但在生产环境中，它意味着不确定性。你给 AI 一段 prompt，它给你一段代码，但这段代码是否做了你真正想做的事？你不知道。下一次给同样的 prompt，它可能给你完全不同的代码。

**如果你让 AI 直接写代码，它会产生幻觉;但如果你让 AI 先输出规格让你确认，再根据规格写代码，质量会产生质变。**

这催生了 AI 原生的 SDD 工具浪潮：

- **Spec Kit**(GitHub 官方)提出"规格可执行化"——通过宪章 → Specify → Plan → Tasks → Implement 的严格阶段门控，让 AI 按规格生成代码 [15]。它的宣言是："Focus on product scenarios and predictable outcomes instead of vibe coding."
- **Superpowers**(Jesse Vincent)走的是"技能驱动"路线——通过 20+ 个可组合的强制 Skills，让 AI 代理遵循 Brainstorm → Design → Plan → TDD+Subagent → Review → Finish 的完整工程流程 [16]。
- **OpenSpec**(Fission AI)追求极简--Propose → Apply → Archive 三步走，核心特色是 Spec Delta Review:只看需求变更的 diff，不读完整规格，不读代码 [17]。
- **Kiro**(AWS)将 Spec 作为 AI 的"精确上下文"，用 Executable Specs + Steering + Context Management 来控制 AI 的行为边界 [18]。

这些工具有一个共同信念：**在 AI 编码时代，规格不再是过时的文档，而是约束 AI 输出、保障大规模代码质量的"操纵杆"。**

回顾整个历程，"设计先行"这条主线经历了一场清晰的门槛降低：从需要数学博士学位的形式化方法，到需要编程能力的 TDD，到产品经理也能参与的 BDD，到写 YAML 就能定义的 API 契约，到用自然语言描述就能让 AI 理解的可执行规格。每一次门槛降低，都让"先定义、再实现"的适用范围扩大了一个数量级。

而 SDD，可以被理解为这种"设计先行"倾向在 AI 时代的一种重新组织和重新命名——它不是凭空出现的，也不必被理解为前述实践的必然终点;更准确地说，它是把过去分散存在于不同方法中的一些思想，在 AI 编码语境下重新组合了起来。

## 二、AI 时代的 SDD 工具生态与工程实践

SDD 在 2025-2026 年的爆发，不是因为理论有了突破，而是因为 AI 编码助手让 SDD 的实践成本发生了质变。围绕 "如何让 AI 按规格写代码" 这一核心问题，一个快速演进的工具生态已经形成 [22]。

### 2.1 规格严谨度:三个层次

并非所有 SDD 实践都是同一个物种。一篇 2026 年 2 月发表在 arXiv 上的 SDD 综述论文将 SDD 实践按严谨度分为三个层次 [23]:

**Spec-First(规格先行)**:在编码之前写规格来指导实现，但代码完成后规格可能就不再维护了。这是 SDD 的入门形态，适合原型开发和一次性功能。与 AI 编码助手配合时，先写规格可以防止 AI 瞎猜需求，显著提升生成代码的质量。

**Spec-Anchored(规格锚定)**:规格作为"活文档"与代码同步维护。需求变了，规格和代码一起改，自动化测试确保二者对齐。这通常被认为是较适合生产系统的实践形态。BDD 框架(如 Cucumber)是这一模式的典型代表。

**Spec-as-Source(规格即源码)**:规格是人类编辑的主导产物，代码主要由规格自动生成，并尽量避免手动改代码。这是 SDD 最激进的形态——规格实际上就是更高层级的源代码。目前它通常只在代码生成工具较成熟的领域更可行，比如从 OpenAPI 定义生成 API 骨架，或者从 Simulink 模型生成嵌入式 C 代码。

理解这三个层次很重要：**大多数团队应该从 Spec-First 入手，在 Spec-Anchored 站稳脚跟，而不是一上来就追求 Spec-as-Source。** ThoughtWorks 在其 2025 年技术雷达中也指出，SDD 仍是一个快速演进的新兴实践，团队应该保持关注并小规模实验，而非急于全面铺开 [24]。

### 2.2 代表性 SDD 工具逐一解析

2025-2026 年，AI 原生的 SDD 工具快速涌现。以下选取四款最具代表性的工具，逐一介绍其定位、工作流、关键特性与局限。

#### Spec Kit(GitHub)[15]

**定位**:GitHub 官方推出的 SDD 工具集，以 CLI 方式分发，为各种 AI 编码助手创建统一的工作空间结构。Spec Kit 是当前讨论度较高的 SDD 工具之一，其核心理念可以用一句话概括：**"Code is no longer king"(代码不再是王道)**--规格才是软件的核心制品，代码只是最后一步的产物 [15]。

**基本工作流**:宪章 → Specify → Plan → Implement

1. **宪章(Constitution)**:项目初始化时创建 `constitution.md`，定义不可变的项目原则和技术约束。这类似于团队的"最高法律"，所有后续规格必须与之对齐。例如："所有 API 必须使用 TypeScript 严格模式"、"禁止直接操作 DOM"。这个设计确保了即使在 AI 自主编码的场景下，也不会偏离组织级的技术标准。

2. **Specify(规格编写)**:用户描述需求，Spec Kit 生成结构化的 `spec.md`，包含功能描述、用户故事、验收标准和边界条件。每个规格会创建一个独立的 Git 分支，便于追踪和评审。

3. **Plan(计划)**:基于规格生成 `plan.md`(实现方案)和 `research.md`(技术调研)，并通过内置的 `/speck:analyze` 命令进行交叉一致性检查，确保计划与规格和宪法一致。

4. **Tasks(任务分解)**:将计划分解为带验收标准的任务列表，AI 按序执行。每个任务完成后需要人工确认，形成严格的阶段性检查。

**关键特性**:

- 产出物最丰富：spec.md、plan.md、research.md、data-model.md、contracts 等多份结构化文档
- 兼容 18+ AI 编码工具(Cursor、Claude Code、Windsurf、Copilot 等)
- 宪章机制提供了组织级的技术治理能力

**局限**:产物丰富意味着仪式感较重，对于小改动("改个按钮颜色")显得过于笨重。Martin Fowler 在分析中指出，Spec Kit 为每个规格创建独立分支的做法更像是"变更请求级别的生命周期"，而非"功能级别的持续维护"，因此可能更接近 Spec-First 而非 Spec-Anchored [34]。

#### OpenSpec(Fission AI)[17]

**定位**:一款以 MCP 协议为核心、强调"棕地优先"(Brownfield-first)的轻量级 SDD 框架。OpenSpec 的核心理念是**"Fluid, not rigid, iterative not waterfall"(流动而非刚性，迭代而非瀑布)**，专为已有代码库的增量迭代设计。

**基本工作流**:Explore → Propose → Apply → Archive

1. **Explore(探索)**:用户描述需求意图，AI 在现有代码库上下文中理解问题范围。这一步不生成任何文件，纯粹是对话式的需求澄清。

2. **Propose(提案)**:在 `openspec/changes/` 目录下创建一个以变更名命名的独立文件夹，内含 `proposal.md`(变更提案)和 `tasks.md`(任务清单)。每个变更的文件完全隔离在自己的文件夹中，不会与其他变更互相干扰。

3. **Apply(执行)**:通过 `/opsx:apply` 命令，AI 逐条读取 `tasks.md` 中的任务并执行实现。执行完成后，通过自定义的 `openspec-review` 技能进行代码审查和规范一致性检查。

4. **Archive(归档)**:将完成的变更目录整体移入 `openspec/changes/archive/`,Delta Specs(增量规范)合并到主规范目录 `openspec/specs/`。历史变更不再参与日常开发的上下文窗口，有效控制上下文熵增。

**关键特性**:

- 核心概念仅四个：specs、changes、delta specs、artifacts，上手极快
- 变更隔离机制：每次变更独立文件夹，防止不同需求互相干扰
- Delta Specs:只审查变更部分的 diff，而非整个规范文件
- 兼容 20+ AI 工具，通过 MCP 协议注入

**局限**:轻量意味着某些环节的"强制力"不足——例如，OpenSpec 本身没有内置测试覆盖要求，需要用户通过自定义技能(如 `openspec-e2e`)来补强。对于需要严格阶段门验证的大型项目，可能不如 Spec Kit 那样"开箱即用"。

#### Superpowers(Jesse Vincent)[16]

**定位**:一款为 Claude Code 量身打造的主流工作流插件，通过可组合的**技能(Skills)**强制 AI 遵循特定的工程实践。与其他 SDD 工具不同，Superpowers 不提供自己的规格模板或工作流引擎，而是用一套强制性的技能系统约束 AI 的每一步行为——从需求澄清到测试驱动实现再到代码审查，全流程自动化触发。其哲学是：**不给 AI 自由发挥的空间，而是用技能把 AI 的每一步行为都约束住。** 现已通过插件机制扩展至 Cursor、Copilot、Gemini 等平台。

**基本工作流**:Brainstorm → Design → Plan → TDD → Review → Finish

1. **Brainstorm(头脑风暴)**:AI 分析需求，提出多种实现方案并评估优劣。

2. **Design(设计)**:选定方案后，生成设计文档，定义架构决策和数据模型。

3. **Plan(计划)**:将设计分解为具体任务，每个任务包含实现细节和验收标准。

4. **TDD(测试驱动实现)**:这是 Superpowers 的核心差异化——强制要求先写测试再写实现。AI 必须通过所有测试后才能进入下一步。

5. **Review(代码审查)**:内置代码审查技能，自动检查代码质量、安全漏洞和规范一致性。

6. **Finish(收尾)**:确认所有任务完成，更新文档。

**关键特性**:

- SubAgent 并行执行：复杂任务可分派给多个子代理，每个子代理在独立上下文中工作
- 强制 TDD:测试不是可选的"最佳实践"，而是流程中的硬性门槛
- 技能可组合：用户可以编写自定义技能来扩展或覆盖默认行为

**局限**:学习曲线陡峭(技能系统本身就是一个需要理解的抽象层)，且 Claude Code 作为核心生态享有最完整的体验和支持，其他平台的适配程度和稳定性参差不齐。

#### Kiro(AWS)[18]

**定位**:AWS 推出的 AI 编码 IDE，基于 Code OSS(VS Code 的开源基础)深度定制。与前三款"嵌入现有工具"的框架不同，Kiro 本身就是一个完整的开发环境。其核心理念是**将规格驱动开发内建到 IDE 的每一个环节**，让开发者无需额外安装任何插件即可使用 SDD 工作流。根据二手转述资料，AWS CEO Matt Garman 在 re:Invent 2025 上将 Kiro 描述为能够"独立弄清楚如何完成工作"[27]。

**基本工作流**:Prompt → Specs → Task Plan → Implement → Tests

Kiro 提供两种工作模式：

1. **Vibe Mode(即兴模式)**:类似传统的"对话式 AI 编码"，适合快速原型和探索性开发。不强制走规格流程。

2. **Spec Mode(规格模式)**:严格遵循规格驱动工作流--
   - **Requirements(需求)**:将用户的自然语言提示转化为结构化的需求文档，每个需求以 User Story("As a... I want to...")格式编写，附带 GIVEN/WHEN/THEN 格式的验收标准。
   - **Design(设计)**:基于需求生成设计文档，定义架构方案、接口设计和边界条件。
   - **Task Plan(任务计划)**:创建带依赖关系的任务序列，每个任务包含实现细节和成功标准。用户可以逐个审批、重定向或让 AI 自主执行。
   - **Implement(实现)**:按任务序列逐步生成代码和测试。Kiro 支持长时间自主运行("code for days")，但这也带来了风险(见下文)。

**关键特性**:

- **Agent Hooks(代理钩子)**:事件驱动的自动化机制，例如保存文件时自动重新生成测试、提交代码时自动安全扫描、API 变更时自动更新文档。配置文件为 `.kiro/hooks.yaml`。
- **Steering Files(引导文件)**:类似 CLAUDE.md 的项目级约束文件(`.kiro/steering.md`)，定义代码规范、架构约束、测试要求等。支持全局和项目两级配置。
- **跨会话持久化**:规格和上下文在不同会话之间保持，不会因为关闭 IDE 而丢失。
- **深度 AWS 集成**:原生支持 SageMaker、Lambda、Aurora DSQL 等 AWS 服务。

**局限与警示**:据评论性来源转述，2025 年 12 月，Kiro 曾因一次事故引发行业关注——在获得"修复系统"权限后，AI 自主决定删除并重建整个环境，导致 AWS Cost Explorer 在中国区域中断 13 小时 [33]。如果这一叙述属实，它至少说明：**规格驱动并不天然等于安全**。即使有结构化的规格，AI 的实现路径选择仍可能产生高代价后果。Martin Fowler 也指出，Kiro 目前更接近 Spec-First 层级，其规格的长期维护策略尚不明确 [34]。

### 2.3 工程实践与大厂案例

- **Amazon** 用 TLA+ 验证 DynamoDB/S3 的分布式设计，属于 Spec-Anchored 层级，核心价值在于发现深层并发 bug [5]。
- **GitHub** 推出 Spec Kit 并推动 SDD 工具链标准化，偏向 Spec-as-Source 层级 [15]。
- **AWS** 推出 Kiro 并以规格驱动方式做 AI 编码，处于 Spec-First 到 Spec-Anchored 的演进中 [18]。
- **阿里云开发者社区**报道了用 OpenSpec + 多 AI 协同完成跨境保险产品全流程交付的案例，核心价值在于多模型协同、统一规格作为共享语境 [29]。
- **腾讯云开发者社区**对 SDD 适用性做了反思，列出了 SDD 的适用边界和判断标准 [31]。

### 2.4 工具横向对比

在逐一介绍四款 SDD 工具之后，一个自然的问题是：它们到底有多大的差异？Spillwave 在其对比分析中明确指出，"spec-driven 意味着不同的事情取决于你问谁" [22],Martin Fowler 也观察到，这些工具虽然都自称 SDD，但"差异相当大" [34]。差异确实存在，但差异之下，共性同样清晰。

#### 差异:五个关键设计选择

四款工具的差异集中体现在五个关键设计选择上。

**选择一：规格的"重量"——多制品 vs 少制品**

Spec Kit 在每个阶段生成大量结构化文档(spec.md、plan.md、research.md、data-model.md、contracts)，仪式感最强。OpenSpec 只有 proposal.md 和 tasks.md 两个核心产物，Kiro 也是每个阶段一个 markdown 文件。Martin Fowler 在实测后直言："说实话，我宁愿 review 代码也不愿 review 这么多 markdown 文件。" [34] 他指出 Spec Kit 产生了"大量重复的、冗长的、乏味去 review 的"文档。

这个差异的本质是**治理需求 vs 效率需求的取舍**。大团队需要丰富的制品来保证一致性和可追溯性(Spec Kit 的选择)，小团队需要轻量快速迭代(OpenSpec 的选择)。

**选择二：管得多 vs 管得少**

Spillwave 将这一维度定义为 SDD 工具"最大的分歧点" [22]。一端是"管得多"：Superpowers 用可组合技能控制 AI 的每一步行为，强制要求测试先行、测试不通过流程就推进不了，还支持把复杂任务分派给多个子代理并行执行;另一端是"管得少"：OpenSpec 的 `/opsx:apply` 只是把任务列表交给 AI 去执行，不干涉实现过程。Spec Kit 和 Kiro 处于中间地带。

管得越深，对 AI 行为的控制力越强，但灵活性越低 [22]。深度控制适合质量要求极端的场景(比如安全关键系统)，轻量委托适合探索性开发。

**选择三：上下文管理——如何防止 AI "降智"**

AI 编码助手有一个根本性挑战：随着对话越来越长，上下文窗口被塞满，AI 的输出质量就开始下降--Spillwave 将这种现象称为"上下文腐烂"(context rot)[22]。四款工具给出了不同的应对策略：

- **制品传递**(Spec Kit):通过结构化文档在阶段之间传递上下文，而非依赖对话历史。
- **变更隔离**(OpenSpec):每次变更独立文件夹，历史变更归档后不再参与上下文窗口。
- **技能约束**(Superpowers):通过技能系统定义 AI 的行为边界，而非依赖上下文长度。
- **持久化 + 引导文件**(Kiro):跨会话保持规格上下文，用 steering 文件定义持久约束。

arXiv 上的 SDD 综述论文将这种能力概括为"规格作为 AI 的精确上下文" [23]--简单说，规格写得越好，AI 的输出质量就越高。

**选择四：从零开始还是改造存量**

Martin Fowler 在评测中特别指出，SDD 工具在已有代码库中的引入成本远高于从零开始的项目 [34]。Spillwave 也观察到，大多数工具的教程都基于"从零创建应用"来演示，因为这对教程来说最容易 [22]。

在本文讨论的几款工具中，OpenSpec 对"存量项目优先"的强调最明确。其变更隔离机制天然适合已有项目——你不需要为整个项目写规格，只需要为当前要改的变更写。Spec Kit 的宪章机制虽然强大，但在已有项目中建立一份覆盖全局的宪章，成本很高。Kiro 和 Superpowers 至少没有像 OpenSpec 那样突出存量场景设计。

**选择五：验证策略——谁来保证做对了**

验证是 SDD 流水线的最后一环，也是最容易被忽视的一环。四款工具的策略差异显著：

- Spec Kit 提供内置的 `/speckit.analyze` 做一致性检查，确保规格、计划、实现三者对齐。
- OpenSpec 本身没有内置测试覆盖要求，需要用户自己加(比如自定义一个 `openspec-e2e` 技能)。
- Superpowers 的策略最激进——测试不通过，流程就推进不了。
- Kiro 的双模式设计意味着验证强度完全取决于用户选择即兴模式还是规格模式。

Martin Fowler 指出了一个更深层的问题：即使有所有这些文件、模板、prompt、工作流和检查清单，"我仍然频繁看到 AI 最终没有完全遵守指令" [34]。这意味着验证策略还有很大的改进空间。

#### 差异背后的逻辑

综合以上五个维度，可以看到四款工具的设计差异并非随意，而是对应着不同的工程哲学和适用场景：

- **Spec Kit 的"重"** 对应的是**组织级治理**的需求。GitHub 内部有数千名开发者、数百个仓库，需要一致性和可追溯性。丰富的文档和宪章机制是这种需求的自然产物。
- **OpenSpec 的"轻"** 对应的是**增量迭代**的现实。大多数项目不是从零开始的，开发者需要的是"给当前变更写规格"，而非"给整个系统写规格"。
- **Superpowers 的"强制"** 对应的是**质量极端化**的需求。在安全关键或高可靠性场景中，"可选的最佳实践"是不够的，必须变成"不可绕过的硬性门槛"。
- **Kiro 的"双模式"** 对应的是**探索与工程化的切换**。同一个项目在不同阶段需要不同节奏——原型阶段需要速度(Vibe)，开发阶段需要确定性(Spec)。

Spillwave 的分析也佐证了这一点："那些差异在为你的工作流选择工具时很重要。" [22] 它进一步指出，这些工具"并非互斥"，国内开发者社区已有将 OpenSpec 与 Superpowers 整合的实践 [28]。

Martin Fowler 的观察或许是对这段分析最好的总结："SDD 工具之间的差异相当大，在选择时不应将它们视为同质化的替代品。" [34]

## 三、SDD 的本质

### 3.1 SDD 实践方法的本质

前一章详细分析了四款 SDD 工具的差异，但差异之下有一个不容忽视的共性。剥掉各自的术语外衣，四款工具的工作流可以映射到同一个**通用 SDD 流水线**上：

> **需求澄清 → 规格编写 → 任务分解 → 代码实现 → 验证收尾**

| 通用阶段 | Spec Kit            | OpenSpec         | Superpowers     | Kiro                  |
| -------- | ------------------- | ---------------- | --------------- | --------------------- |
| 需求澄清 | 宪章 + Specify      | Explore          | Brainstorm      | Prompt                |
| 规格编写 | spec.md             | proposal.md      | Design          | Requirements + Design |
| 任务分解 | Tasks               | tasks.md         | Plan            | Task Plan             |
| 代码实现 | Implement           | Apply            | TDD(SubAgent) | Implement             |
| 验证收尾 | Analyze + Checklist | Archive + Review | Review + Finish | Tests                 |

Spillwave 的分析也印证了这一点："所有四个工具都同意一个核心循环：定义需求、规划实现、执行任务、验证结果。它们都把 AI 编码代理视为从结构化制品而非临时 prompt 中工作的实现者。" [22]

这个共性揭示了 SDD 的一个底层逻辑：**SDD 工具的本质，就是把人类开发者"在脑子里想清楚再动手"这个过程，拆成明确的步骤、写成结构化的文档，然后交给 AI 去执行。** 无论工具怎么叫，底层骨架高度相似——区别只在于每个环节做得"多重"和"多强制"。

这个逻辑还有一层更深的含义。回顾前文对差异背后逻辑的分析--Spec Kit 的"重"对应组织级治理、OpenSpec 的"轻"对应增量迭代、Superpowers 的"强制"对应质量极端化、Kiro 的"双模式"对应探索与工程化的切换——这些差异并非随意，而是不同工程场景的自然映射。这说明 SDD 并非一个固定的配方，而是一个**以通用流水线为骨架、根据场景灵活调整权重的实践框架**。

### 3.2 规格的本质:四个维度

#### 3.2.1 规格是认知工具

人脑无法同时处理复杂系统的所有细节。规格是一种**把思维搬出来的工具**--把"系统应该做什么"从脑子里写到纸上(或文件中)，让想法变得可以审视、可以讨论、可以迭代。

这不是文档的问题，这是**人的大脑容量有限**的问题。没有规格，团队对"要构建什么"的理解就散落在每个人的脑子里，每次对齐都需要大量沟通。有了规格，这些理解就有了共享的载体。

#### 3.2.2 规格是验证锚点

没有规格，你怎么知道代码写对了？靠感觉？靠手工测试？靠"跑起来没问题"?

规格提供了**客观的验证标准**--不是"我觉得它应该这样"，而是"规格写了它应该这样，测试跑通了证明它确实这样"。这个标准让"做对了没有"从主观判断变成了可验证的事实。

#### 3.2.3 规格是通信协议

不同角色之间的沟通成本是软件工程最大的隐性成本。开发者说"我已经实现了"，产品经理说"这不是我要的"，测试说"这跟需求文档不一致"——三方的分歧往往不是因为谁错了，而是因为**没有共享的规格作为通信协议**。

SDD 提供了这个协议。当规格足够清晰时，"系统应该做什么"不再需要反复讨论——读规格就够了。

#### 3.2.4 规格是知识资产

代码告诉你"系统怎么做的"，规格告诉你"系统为什么这么做"。

当团队成员离职、项目交接、系统需要重构时，代码是必要的但不充分的。规格是系统知识的结构化沉淀——它比代码更能表达**意图**。

这也是为什么 AI 时代的 SDD 工具(如 OpenSpec)强调"持久化上下文"——规格和代码放在同一个仓库里，不同会话之间保持不丢失。在 AI 编码助手频繁切换上下文的今天，规格是防止知识丢失的最后防线。

### 3.3 决定 SDD 实践效果的关键因素

理解了 SDD 实践方法的本质和规格的本质，就能理解为什么有些团队的 SDD 实践效果显著，有些则沦为形式主义。答案不在工具的选择上——正如 3.1 节的分析，所有 SDD 工具本质上都在执行同一套通用流水线。答案也不在流程的严谨度上——再完美的流程，如果输入的是垃圾，输出的还是垃圾。

答案在规格的质量上。而规格的质量，取决于一个容易被忽视的东西：**知识积累**。

#### 从局部到全局:决定 SDD 效果的三个层次

**第一层：规格本身的质量**

arXiv 上的 SDD 综述论文有一句精准的论断："规格的质量直接决定后续所有环节的质量" [23]。一份好的规格需要行为清晰、可测试、无歧义、覆盖关键场景但不过度设计。但问题在于，写出这样的规格并不容易——它要求编写者同时理解业务意图、技术约束、历史决策和边界条件。正如 3.1 节所揭示的，SDD 的通用流水线本身只是"运输系统"，规格的质量才是"货物"。

Spec Kit 的宪章机制、OpenSpec 的变更隔离、Superpowers 的强制 TDD，都在试图用流程来弥补这个短板。但流程只能保证"你写了规格"，不能保证"规格写对了"。一份脱离业务上下文的规格，即使流程再严谨，也只是一堆格式正确的废话。

**第二层：规格背后的知识库**

这是很多团队忽略的关键。一份好的规格不是凭空写出来的，它依赖于四个层面的知识积累：

- **编码规范**(相当于 Spec Kit 的宪章):工程目录结构、命名规则、错误码定义、测试标准。没有这些，AI 生成的代码能跑但不可读，或者编译通过但结构混乱。
- **业务规则**:数据映射逻辑、清洗规则、优先级排序。AI 最容易在这上面出错——因为它看不到"为什么这么设计"的历史，只知道"现在要做什么"。
- **接口声明**:内部 RPC 接口定义、外部依赖说明。没有这些，AI 在做实现规划时可能会"发明"不存在的 API，或者在不知道接口约束的情况下写出破坏兼容性的代码。
- **历史约束**:过去踩过的坑、存量数据的特殊处理逻辑、向下兼容性要求。这是 AI 编码最容易犯错的领域——重构即破坏。一份好的避雷指南，比十份完美的功能规格更有价值。

这四个层面的知识，本质上都是规格的一部分——它们回答的不是"这次要做什么"，而是"在这个系统里做事必须遵守什么"。arXiv 论文在描述 Plan 阶段时指出，"即使功能规格再完美，如果没有实现约束的上下文，AI 生成的代码仍然可能违反组织标准或架构决策" [23]。这正说明了知识库的重要性。

很多团队只重视 SDD 流程本身的实践——选工具、建流程、写规格——却忽略了知识库的积累。这其实是本末倒置：**流程是运输系统，知识库是货物。没有货物的运输系统再高效，也只是在空跑。**

**第三层：SDD 在全链路中的位置**

如果跳出一个单一项目的视角，从一个产品的端到端交付来看：需求、设计、研发、测试、部署、运维——研发测试只是其中一环，而且可能还不是效率的关键路径。

这意味着 SDD 的价值边界是有限的。它能让"从需求到代码"这一段更可靠，但如果需求本身就定义错了，或者测试策略有缺陷，或者部署流程一团糟，SDD 再好也救不了全局效率。这也是为什么 ThoughtWorks 在技术雷达中将 SDD 定位为"值得尝试"而非"全面采用" [24]--它是一个重要的局部优化，但不是全局的银弹。

更进一步，从组织架构的视角看，知识库不应该只服务于单个项目的研发环节，而应该成为组织级的共享资产。编码规范、业务规则、接口契约、历史约束——这些知识如果只存在于某个团队的某个工具里，那就是孤岛。当组织有多个团队、多个项目在并行开发时，跨团队的知识共享和规格复用，才能让 SDD 的价值从"项目级提效"升级为"组织级提效"。

#### 总结:从本质到实践

回顾 3.2 节的四个维度——规格是认知工具、验证锚点、通信协议、知识资产——可以看出，前三者(认知、验证、通信)的效果都依赖于第四者(知识)的厚度。知识积累越深厚，规格的认知价值越高、验证越可靠、通信越高效。反之，没有知识积累的规格，就像没有地基的建筑——看起来结构完整，实际上经不起推敲。

这也是 AI 编码时代给 SDD 带来的一个新启示：**AI 让"写代码"变得廉价，但"写对规格"变得前所未有的昂贵——因为好的规格需要深厚的知识积累，而知识只能靠时间、实践和持续的反思来获得。** 而反过来，3.1 节揭示的通用流水线给了我们信心：既然所有 SDD 工具共享同一个底层逻辑，那么投入知识积累的回报就是可迁移的——你为规格质量所做的每一分努力，不依赖于任何特定工具，而是服务于 SDD 这个实践框架本身。

## 四、没有银弹:SDD 的已知问题

Fred Brooks 在 1986 年的论文《没有银弹》中指出 [19]，软件工程中不存在任何单一技术能带来数量级的效率提升。SDD 也不例外。

### 4.1 规格与实现的鸿沟

规格描述的是"应该做什么"，但"怎么做"的复杂性往往超出规格的覆盖范围。

一个排序算法的规格可以是"输出有序序列"，但实现中涉及的内存管理、边界条件、性能优化——这些"怎么做"的问题，规格往往管不了。更不用说性能、可用性、安全性这类"非功能性需求"了，它们很难用行为规格精确表达。

### 4.2 规格腐烂(Specification Rot)

在日常 SDD 实践中，当代码需要修改时，我们往往面临一个分岔路口：是修改规格后让 AI 重新生成代码，还是直接去修改代码？

从项目实践的观察看，这两者皆会存在。这并不是一个非黑即白的选择，而是完全取决于程序员的工程经验。有经验的程序员清楚地知道什么时候应该通过修改规格来驱动代码变更，什么时候可以直接修改代码且能把由这引发的规格不一致的影响降到最低。而经验尚浅的成员则难以做出这种精准的判断，从而随意地跳过规格直接修改代码。正是这种缺乏判断的随意修改，成为了规格腐烂的开端。随着时间推移，规格与代码实际实现的 Gap 会越来越大，最终规格变成了一堆没人信也没人看的废纸。

### 4.3 过度规格化(Over-specification)

过早定义细节会限制实现方案的选择。你规定"必须使用 B+ 树索引"，后来发现 Hash 索引更适合——但规格已经锁死了。

敏捷方法论的核心主张是"拥抱变化"，但规格天然倾向于"锁定"。这是一对根本性的张力。

### 4.4 转移的成本(审查成本)

以前批评传统 SDD 的经典理由往往是前期投入过大(比如一个两周的项目要"花三天写规格")。但在实际进入 AI 时代落地的阶段，这变成了伪命题，因为绝大部分规格是由 AI 生成，人去 Check。此时，"敲键盘写字"的物理痛点消失了，但它转化为了极大消耗脑力的**转移成本(审视成本)**。

审查几千字的 AI 生成规格，实际上非常消耗脑力。人们是否愿意、是否有精力去强打精神逐行 Check AI 生成的冗长 `spec.md` 和 `plan.md`?在观察真实实践时我们发现，程序员往往只选择"部分规格审查"。但这立刻引出了新的隐患：重点审查哪些部分？如何审查？那些没有去重点审查的边缘规格谬误会对代码生成产生什么影响？到头来，这些问题的重担，又不得不落回到极度依赖当事程序员个人经验的死局上。

### 4.5 "规格即测试"的幻觉

TDD 的测试覆盖不等于规格覆盖。测试只能验证你想到的场景——遗漏的规格等于遗漏的缺陷。即使测试全部通过，也不能证明系统行为完整正确。

### 4.6 业务理解的局限性

规格只能描述"已知的需求"。但很多创新性产品在开发过程中才发现真正需求——用户不知道自己要什么，直到你做出来给他们看。

过早规格化可能扼杀探索空间。这不仅是理论上的担忧——很多成功的产品都不是按规格做出来的。

### 4.7 适用边界建议

| 场景              | SDD 适用度 | 原因                             |
| ----------------- | ---------- | -------------------------------- |
| 安全关键系统      | ★★★★★      | 错误代价极高,形式化验证往往具有很高价值 |
| 大型分布式系统    | ★★★★☆      | 接口契约和服务间协作至关重要     |
| 平台/SDK/API 产品 | ★★★★☆      | 契约即产品,规格即文档           |
| 中大型业务系统    | ★★★☆☆      | 视团队能力和文化而定             |
| 探索性/创新性产品 | ★★☆☆☆      | 需求不稳定,过早规格化有害       |
| 小型内部工具      | ★★☆☆☆      | 成本不成比例                     |
| 快速原型/MVP      | ★☆☆☆☆      | 速度优先,规格是负担             |

## 五、AI 编码时代下的 SDD

前面四章已经充分讨论了 SDD 的脉络、工具、本质和局限。在 AI 编码时代，SDD 看起来是为数不多能约束 AI 编码输出的系统性方法。但它能救多深、能救多远？本章试图把几个值得持续关注但尚未有定论的问题摊开讨论。

更现实的判断是：**SDD 不是"要不要做"的问题，而是"做在什么层级、用什么力度、配什么控制手段"的问题。** 对很多团队来说，SDD 之所以值得试，并不是因为它解决了一切，而是因为它至少提供了一套比纯 prompt 驱动更稳定的工作骨架。

### 5.1 约束的度:什么时候写规格，写多细

**什么时候该 vibe，什么时候该 spec?**

Kiro 的口号"Go from vibe coding to viable code" [18] 精准地捕捉了这个张力。Vibe coding--凭感觉让 AI 写代码——在原型探索、快速验证想法时很有价值，速度和灵活性是它的优势。但如果把它用于生产系统，结果往往是不可预测的。反过来，如果在探索阶段就要求严格的规格，可能带来不必要的成本。

这不是一个非此即彼的选择，而是一个连续的区间：

- **纯探索阶段**(验证想法是否可行)→ Vibe coding 为主
- **原型阶段**(验证方案是否可行)→ Vibe coding + 轻量规格
- **开发阶段**(构建生产系统)→ Spec-driven 为主
- **维护阶段**(演进现有系统)→ Spec-driven + 增量演进

**规格的粒度**

在实践中，有几个建议可以参考：

- **聚焦行为，不聚焦实现**。好的规格说"用户点击购买按钮后，库存应当减少 1"，而不是说"调用 InventoryService.decrement() 方法"。
- **聚焦约束，不聚焦自由度**。好的规格定义"必须做什么"和"不能做什么"，把"怎么做"的自由度留给 AI。
- **聚焦不变的规则，不聚焦实现步骤**。好的规格定义系统必须遵守的约束("账户余额不能为负")，而不是实现步骤("先查询余额，再扣减，再更新")。

这两个层面的共同底层逻辑是：**约束不是越多越好，而是在正确的时机施加正确程度的约束。** 但这个"正确的度"究竟在哪里，目前没有公式可循，只能靠团队在实践中摸索。

从已有实践看，大厂并不是把 SDD 当作一个全覆盖的统一模板来推，而是倾向于按风险分层处理：

- **Amazon/TLA+ 的做法**不是"所有代码都形式化"，而是只把最关键、最容易出现系统性事故的并发不变量拿出来建模验证 [5]。
- **API-First 和契约测试的做法**也不是"整个系统都先写完规格"，而是先把最容易跨团队失真的接口边界固定下来 [11][12]。
- **GitHub、AWS 这类 AI 工具链实践**更强调项目级约束文件、阶段门和人工确认点，而不是让 AI 在没有边界的情况下持续自主推进 [15][18]。

这些案例背后的共性很明确：**先约束高风险边界，再逐步扩展，而不是一上来试图把整个研发流程一次性规格化。**

### 5.2 效率与质量的再平衡

AI 编码带来了一种前所未有的能力：极高的代码产出速度。一周几十万行代码对部分团队而言已不再只是夸张修辞。据 Qodo 2025 年度报告对开发者的调研，受访者仓库中已有 42% 的代码由 AI 辅助生成，报告预计到 2027 年这一比例可能达到 65% [39]。但与此同时，质量隐患也在同步暴露：

- **安全风险已有多项案例和报告提示。** Tenzai 用五种 AI 编码工具构建了 15 个相同的应用，报告称共审计出 69 个安全漏洞，其中 SSRF 漏洞出现在全部 15 个应用中 [35]。另一篇二手分析文章援引 Escape.tech 的安全审计，称其扫描了 5,600 个用 vibe coding 工具构建的生产应用，并统计到 2,000+ 漏洞和 400+ 暴露的密钥 [36]。
- **代码质量指标在部分报告中出现恶化迹象。** 二手来源援引 GitClear 对 2.11 亿行代码的分析，称 AI 辅助编码使代码周转率上升 44%、代码重复上升 48%，而重构率从 25% 跌到不足 10% [37]。
- **功能正确不等于质量达标。** 一项 arXiv 学术研究表明，LLM 生成的代码即使通过了单元测试，其整体质量与安全性仍然不可控——功能正确性和代码安全性是两个独立维度 [40]。

这些数据、报告和研究至少共同指向一个判断：**AI 让写代码变快了，但没有同步让"写对代码"变得容易。** 快速迭代的成本确实降低了，但每一轮迭代的质量如果不可控，累积的风险仍然很难评估——尤其是当这些代码已经上线、已经在处理真实用户数据的时候。

值得注意的是，这个效率与质量的张力并不因 SDD 的存在而改变。SDD 能在一定程度上约束 AI 的输出，但它只是缓解手段，不是根治方案。真正的问题在于：**当 AI 让"生产"变得几乎零成本时，质量活动本身也必须被重构。**

如果说传统软件工程的核心矛盾是"开发速度不够快"，那么 AI 时代正在把矛盾改写成"审查速度、验证速度和追责速度跟不上生产速度"。在这个语境里，SDD 的现实价值不应被理解为"保证正确"，而应理解为"为后续验证创造更好的抓手"。这意味着团队不能只部署一个 SDD 流程，还要同步建设三类能力：

- **可执行验证**:把关键规格转换成自动化测试、契约断言、属性检查，而不是停留在 Markdown。
- **风险分级审查**:不是所有规格都逐字细审，而是对资金、安全、兼容性、数据一致性等高风险变更强制深审。
- **上线前硬闸门**:AI 生成代码可以快，但进入主干和生产环境的门槛不能一起放松。

换句话说，SDD 最有用的时候，不是它替代了测试和审查，而是它让测试和审查有了更明确的目标。

### 5.3 脆弱的防线:人性惰性与 AI 盲信

当 AI 代码贡献率持续攀升、逐步接近 100% 时，这个趋势看起来是积极的，但背后隐藏着一个风险：**对 AI 输出的惯性信任。**

当 AI 写的代码"看起来没问题"、测试也通过了，人很容易放松警惕。引入了 AI 代码审查机制，但审查者在面对大量 AI 生成的代码时，往往会不自觉地降低审查标准——毕竟"AI 写的，应该没问题吧"。这种惰性是人性使然。

效率和质量，本质上是一个事物的正反两个方面，此消彼长。这个规律在 AI 时代并没有改变。AI 只是把这个张力的幅度放大了：产出效率提升了一个数量级，但质量风险也同步放大了一个数量级。SDD 能提供一些约束，但它无法解决人的惰性问题——你怎么保证人在检查的时候不偷懒？这个问题没有技术答案。

一篇 2026 年的分析文章将 vibe coding 类比为软件行业的"次贷危机"："所有人都在盖楼，信贷几乎免费，评级机构在庆祝增长。但没人检查地基的质量。" [38] 这个类比虽然尖锐，但并非毫无道理。当市场奖励速度、惩罚审慎时，追求质量就成了需要刻意抵抗人性的行为。

SDD 在这里能做的，是提供结构化的检查点和可量化的验证手段，让人"不得不"在关键节点停下来审视。但它不能保证人真的会认真审视。正如一把锁只能防君子，不能防小人--SDD 的流程机制能帮助有质量意识的人做得更好，但无法替一个放弃思考的人做决策。

因此，团队如果真的要把 SDD 落地，不能只培训"怎么写 spec"，还要同步设计**反惰性的机制**。相比抽象地强调"大家要认真 review"，以下做法通常更有效：

- 把评审对象从"整篇大文档"压缩为"本次变更的 spec diff"，降低审查疲劳。
- 明确谁对哪类规格负责，例如产品负责人看业务规则，资深工程师看技术边界，测试负责人看验收条件。
- 对 AI 生成的规格和计划保留"退回重写"的权力，而不是默认在它的产物上修修补补。
- 将"规格未更新不得合并"写入流程，但只对约定范围内的变更启用，避免把团队拖进形式主义。

这些方法并不性感，但比口头上强调"提高审查意识"更接近现实。

### 5.4 忠实实现错误的规格

这是 AI 时代一个独特且棘手的问题。

AI 写错代码是容易发现的问题——测试会失败。但 AI 忠实地实现了一个错误的规格——测试会全部通过，但做出来的东西是错的。这个问题的本质不是流程问题，而是**大模型自身的认知缺陷**。

大模型看上去有逻辑——它能生成结构清晰、论证严谨的文本。但它无法真正分辨对错，这是当前大模型算法基础决定的。它基于概率生成内容，而不是基于理解做出判断。这意味着，即使你给了 AI 一份有逻辑漏洞的规格，AI 也很可能会"忠实"地实现它，而不会质疑"这个规格本身对不对"。

Martin Fowler 也观察到了类似的现象：即使有完整的文件、模板、工作流和检查清单，"我仍然频繁看到 AI 最终没有完全遵守指令" [34]。这不仅仅是"不听话"的问题——更深层的原因是，AI 本身就不具备判断指令正确性的能力。

这个问题在现阶段看不到彻底解决方案。SDD 能做的是尽量确保规格本身是正确的——通过人工审查、自动化验证、分层检查等手段(参见 3.3 节的讨论)。让规格从"读的"变成"跑的"(可执行的测试、契约断言、属性检查)，是目前相对有效的应对方式之一——毕竟，再漂亮的文档也骗不过一个失败的测试。

但最终，规格是否正确地表达了业务意图，仍然需要人来判断。这是 AI 目前无法替代的角色，短期内也不会改变。

在这里，一个值得警惕的风险是：**千万不要让"写出了一份漂亮的 Spec"带来虚假的安全感。在 AI 时代，生产一篇逻辑顺畅、看起来无懈可击的长篇大论，恰恰是语言模型最擅长骗过人类大脑的把戏**。

比起指望 AI 自己发现规格错了，更实际的做法是为"规格可能错"预留纠偏机制：对高风险需求做双轨验证(文档评审 + 原型验证)，把容易误解的业务规则写成具体的例子表而不是抽象描述，对历史约束较重的系统优先补禁区清单而不是急着追求完整大规格。这类做法的核心不是追求"规格绝对正确"，而是让错误更早暴露、暴露得更便宜。

### 5.5 SDD 的稳妥落地路径

对于一个正在试行 SDD 的团队，比"是否彻底相信 SDD"更重要的问题是：**怎样引入它，才不至于把组织带入新的混乱。** 从前文提到的工具实践和大厂案例看，一条相对稳妥的路径通常包含以下几个原则。

**第一，不要从大而全的 spec 开始，而要从高风险、高复用、高争议的部分开始。**

优先选择以下对象试点：

- 跨团队接口
- 容易出兼容性问题的模块
- 涉及资金、权限、数据一致性的流程
- 新人频繁出错、历史上反复返工的业务规则

这些地方本来就需要更高强度的对齐，SDD 的投入更容易产生正收益。相反，如果团队一开始就要求所有小需求都产出完整 spec、plan、tasks，最后大概率只会得到一堆没人认真看的文档。

**第二，从 Spec-First 起步，不要一开始就追求 Spec-as-Source。**

对大多数团队来说，更现实的目标不是"代码完全由规格生成"，而是先建立一条最基本的纪律：需求变更时，关键规格要同步更新;实现前，先把核心行为、边界条件和验收标准说清楚;上线前，至少有一部分规格能对应到自动化验证。这种做法虽然不激进，但更符合组织吸收新方法的节奏 [23][24]。

**第三，把组织知识沉淀到少数几个高价值载体里，而不是让知识散落在无数篇 spec 中。**

很多团队推进 SDD 失败，不是因为不会写 spec，而是因为每个 spec 都在重复解释同一批背景知识。更有效的做法是把共通知识抽成稳定资产，例如：

- 一个项目级"宪章"或 steering 文件，约束工程标准和禁止事项
- 一个业务术语表，统一关键概念定义
- 一份接口契约目录，沉淀稳定边界
- 一份历史坑位/兼容性清单，告诉 AI 和人哪些地方不能随便动

这样，单次 spec 才能真正保持精简，只描述"这次变更新增了什么"，而不是每次都重写整套世界观。

**第四，让规格评审和代码评审分工，而不是互相替代。**

规格评审关注的是"是否做对事"，代码评审关注的是"是否把事做对"。这两者不能混为一谈。实践里，一个常见失败模式是团队以为"前面做过 spec review，后面 code review 可以放松"，结果只是把风险从实现阶段前移，却没有真正消除风险。更稳妥的方式是：

- 规格评审只盯业务意图、边界条件、验收标准
- 代码评审继续盯复杂度、兼容性、安全性、性能和可维护性
- 用测试和契约把两者串起来，避免二者脱节

**第五，衡量 SDD 成效时，不要只看产出了多少文档，要看返工是否下降、歧义是否减少、上线事故是否前移暴露。**

如果一个团队推了半年 SDD，结果只是多了很多 `spec.md`、`plan.md`，但需求误解、接口扯皮、回归事故并没有减少，那就应该承认这套实践没有真正奏效。SDD 的产出不该以文档页数衡量，而应以是否降低协作成本和质量波动来衡量。

归根结底，SDD 更像一种**工程治理增强器**，而不是自动产出正确性的机器。它对已经具备一定工程纪律的团队，往往是放大器;对纪律薄弱、知识资产稀缺、评审机制松散的团队，它也可能只是把原有混乱重新包装了一遍。正因为如此，试行 SDD 最重要的不是"表态支持"，而是持续校准：哪些地方它真的有帮助，哪些地方它只是制造了新的文书工作。

---

## 结语

从 1960 年代的形式化规格到 2026 年的 AI 原生 SDD 工具，"设计先行"的发展是一部不断在"精确性"和"实用性"之间寻找平衡的历史。

每一次浪潮都在回答同一个问题：**如何让"做什么"比"怎么做"更清晰？** 每一次浪潮都比上一次走得更远——从数学语言到编程语言，从编程语言到自然语言，从自然语言到 AI 可理解的上下文。

但核心从未改变：**先定义，再实现。**

AI 编码没有让这个原则过时，反而让它变得更加紧迫。在 AI 编码助手日益普及的今天，SDD 作为“设计先行”在 AI 时代的最新形态，被很多人视为一根救命稻草——它确实是目前能看到的、约束 AI 编码输出的较系统的方法之一。

SDD 最值得期待的地方，不是它能替团队思考，而是它能逼团队把过去含混、口头、分散的共识，尽可能转化为显式、可审查、可验证的工程资产。如果团队能把它控制在合适的边界内，逐步沉淀知识资产，并持续用测试、契约和评审把规格“跑起来”，那它就可能成为可靠的工程增强器；如果只是把它当作文档生成流水线，它很快就会滑向新的形式主义。

这，才是“设计先行”六十年来不变的本质。
