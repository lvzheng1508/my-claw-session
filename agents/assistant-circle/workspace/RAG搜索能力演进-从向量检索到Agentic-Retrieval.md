# RAG 搜索能力演进：从向量检索到 Agentic RAG

> 当 AI 开始像人一样搜索，RAG 的含义就彻底变了。

## 引言

2020 年，Lewis 等人在论文 *Retrieval-Augmented Generation for Knowledge-Intensive NLP Tasks* 中提出了一个简洁的架构：用一个 Retriever 从外部知识库检索相关文档，再由一个 Generator 基于检索结果生成回答。这就是 RAG 的起点——一个 Retriever 加一个 Generator 的双塔结构 [1]。

六年后的今天，伴随大模型基础能力的爆发，RAG 的内涵已经远远超出了单纯的“向量检索 + LLM 生成”。

2026 年 3 月，微软在其官方文档中重新定义了 RAG："RAG 不再默认等于向量搜索，而是任何形式的外部知识增强生成。" [2] 同月，学术界发布了首篇 Agentic RAG 系统综述，将其形式化为“有限视野的部分可观察马尔可夫决策过程” [3]。而 Anthropic 在其工程博客中提出了一个更具启发性的观点——“Seeing like an Agent”：搜索应该变成一种渐进式的发现（progressive disclosure），而不是一次性的检索 [4]。

这条演进线的核心脉络是什么？**搜索的主动权，正在越来越多地从系统预定义转向模型主导。**

本文将从三个维度展开这条脉络：

1. **为什么向量检索不够了**——单一检索模式的根本局限
2. **搜索能力的扩展**——Graph RAG、Hybrid Search、Agentic RAG
3. **未来走向**——混合检索是务实选择，Agentic RAG 值得投入关注，但落地前提比概念更重要

## 一、向量检索：好用但不够用

### 1.1 Naive RAG 的黄金时代

早期的 RAG 实现几乎千篇一律：将文档切分成 chunk，用 embedding（词嵌入，即将文本转化为高维浮点向量）模型向量化，存入向量数据库，查询时计算语义相似度，取 top-k 结果拼接到 prompt 中。这套流程被称为 **Naive RAG**。

> **通俗理解**：这就好比一场开卷考试。无论遇到什么题，检索系统都是不管三七二十一给你翻出最像的 5 页书，然后让你（大模型）根据这 5 页内容去抄答案。

在知识密集型问答场景中，Naive RAG 确实立竿见影。Lewis 等人的实验表明，RAG 模型在三个开放式 QA 任务上达到了当时的 SOTA，生成的语言比纯参数化模型更具体、更多样、更符合事实 [1]。

但问题很快就暴露了。

### 1.2 向量检索的五重局限

**局限一：语义漂移**

向量检索基于语义相似度，但“语义相近”不等于“内容相关”。查询“Java 性能优化”可能召回“JavaScript 性能优化”的文档——两者向量距离很近，但对用户毫无价值。

**局限二：精确匹配缺失**

向量检索在处理精确匹配场景时表现差强人意：错误码（如 `ERR_CONNECTION_REFUSED`）、产品 ID、专有名词、日期等。这些内容在向量空间中缺乏足够的语义区分度。

Azure AI Search 的技术文档明确指出了这一点："某些场景——如查询产品代码、高度专业化术语、日期和人名——关键词搜索因为能识别精确匹配而表现更好。" [5]

**局限三：实体关系缺失**

向量是扁平的。它知道“张三”和“李四”各自是谁，但不知道“张三认识李四”。这种实体间的关系信息在纯向量空间中无法有效表达。

这正是微软在 2024 年提出 GraphRAG 的直接动机 [6]。

**局限四：数据同步成本高**

每次文档变更，都需要重新执行 chunk → embed → index 的全链路。对于频繁更新的知识库，这个成本不可忽视。更棘手的是，增量更新在向量数据库中并不总是可靠的——新旧 chunk 之间的语义边界可能导致检索结果的不一致。

**局限五：Chunk 边界问题**

文档切分本身就是一个信息损失的过程。一个跨越 chunk 边界的论述，其上下文会被切断，导致检索到的片段缺乏完整语义。无论用固定长度、按段落还是按语义切分，这个问题都无法完全消除。

### 1.3 社区的应对

面对这些局限，社区并没有停留在抱怨层面，而是沿着两条路径推进：

- **路径 A**：在向量检索的基础上修补——加入 re-ranking、query rewriting、multi-hop retrieval 等机制。这被称为 **Advanced RAG** [7]。
- **路径 B**：重新思考检索本身的形态——引入图数据库、关键词搜索、结构化查询，甚至让模型自己决定怎么搜。这最终导向了 **Agentic RAG**。

## 二、搜索能力的扩展

### 2.1 Advanced RAG：在向量检索基础上修补

Advanced RAG 本质上是对 Naive RAG 的“打补丁”。核心改进包括：

**Query Rewriting（查询改写）**：将用户的原始查询改写为更适合检索的形式。例如，将模糊的"这个 bug 怎么修"改写为具体的错误描述。

**Re-ranking（重排序）**：先用向量检索召回较多候选(如 top-50)，再用一个更精确的模型（如 cross-encoder，即交叉编码器，能同时理解查询和文档的相关性）对结果重新排序。Cohere Rerank 是这一方向的代表产品。

**Multi-hop Retrieval（多跳检索）**：对于需要多步推理的复杂问题，执行多次检索并综合结果。例如回答“A 公司的 CEO 的母校在哪里？”需要先找到 CEO 是谁，再查找其教育背景。

CRAG（Corrective RAG）[8] 提出了更进一步的思路：在检索后增加一个质量评估环节，如果检索结果的置信度低于阈值，就触发 web 搜索来补充。这本质上是给检索加了一个“纠错机制”。

这些改进确实提升了效果，但有一个根本问题没有解决：**检索策略是预定义的，模型无法根据具体任务动态调整。**

### 2.2 Graph RAG：补上关系推理

2024 年 4 月，微软研究院发布了 GraphRAG 论文 [6]，核心思路是：用 LLM 从文档中抽取实体和关系，构建知识图谱，再预生成社区摘要。查询时，基于社区摘要生成局部回答，最终汇总为全局回答。

> **通俗理解**：如果说前面的向量检索是“翻字典查包含某句话的段落”，那图检索就是“理家谱、画人际关系网”。它不仅知道谁是谁，更擅长回答“张三的合伙人昨天投资了哪个项目”这种盘根错节的问题。

GraphRAG 解决了一个传统 RAG 难以应对的问题——**全局性问答**。比如“这个数据集的主要主题是什么？”这类需要对整个语料进行归纳的问题，传统 RAG 几乎无能为力，而 GraphRAG 通过社区摘要可以给出全面且有深度的回答。

但 GraphRAG 的局限不只是“有待改进”——对大多数团队来说，它是一道门槛：

- **建图成本极高**：需要对每个文档进行 LLM 实体抽取，API 调用量惊人。对于一个中等规模的企业知识库(数千篇文档)，建图过程可能消耗数百万 token 的 LLM 调用，时间和金钱成本都远超传统向量索引。这直接将 GraphRAG 推向了大预算团队才玩得起的位置。
- **批处理模式**：原始 GraphRAG 采用离线批处理建图，数据更新后需要全量或大批量重建。对于频繁变更的知识库(如每日更新的技术文档)，这意味着建图过程变成了持续的成本中心。
- **查询延迟高**：社区摘要的生成和汇总通常需要秒级到十秒级的延迟，不适合对实时性要求高的场景。
- **质量不可控**：实体抽取和关系识别完全依赖 LLM，抽取质量会因文档类型、领域复杂度、prompt 设计等因素波动，且缺乏直观的质量评估手段。
- **幻觉固化进存储层**：这是 GraphRAG 最容易被忽略、也是性质最严重的问题。传统 RAG 中，模型的幻觉发生在生成阶段——你可以通过检索结果验证来纠偏。但 GraphRAG 在建图阶段就调用了 LLM 进行实体抽取和关系识别，如果 LLM 在这一步产生了幻觉（如捏造了不存在的实体关系、错误地归因了事件因果），这些错误会被**直接写入知识图谱并持久化**。后续所有查询都会基于这个被污染的图谱进行推理，而用户完全无法感知。“建图成本高”是工程问题，可以用钱解决；“幻觉固化进存储层”是方法论缺陷，会系统性污染所有后续检索。两者的性质完全不同。这个问题在当前讨论中被严重低估了。

Graphiti [9] 试图解决其中部分问题。它引入了**时序上下文图**的概念：每个事实都有一个**有效性窗口**（`valid_at` / `invalid_at`），信息变更时旧事实不会被删除而是标记失效，支持"现在是什么情况"和"过去某个时间点是什么情况"的查询。它还支持**增量图构建**，理论上可以避免全量重算。

但需要注意，Graphiti 目前仍是一个相对年轻的开源项目。其增量构建在中等规模数据集上表现良好，但在企业级生产环境（百万级节点、高频更新、多租户隔离）中的稳定性和性能表现，尚未有充分的公开验证。将 Graphiti 的官方描述直接等同于生产级承诺，是不够审慎的。

### 2.3 Hybrid Search：混合检索正在成为主流工程选择

单一检索模式的局限已经得到业界广泛认可。Azure AI Search、Elasticsearch、Pinecone 等主流搜索平台，也都开始将混合检索作为更常见的默认能力。

Azure AI Search 的实现方式最具代表性 [5]：

- **单请求执行**：一个查询同时触发全文搜索（BM25，基于词频的经典排序算法）和向量搜索（HNSW，近似最近邻索引算法）
- **并行执行**：两种搜索同时进行，不增加顺序延迟
- **RRF 合并**：用 Reciprocal Rank Fusion（倒数排名融合）算法合并两路结果

```json
{
  "search": "historic hotel walk to restaurants and shopping",
  "vectorQueries": [
    {
      "kind": "vector",
      "vector": [
        /* embedding */
      ],
      "k": 50,
      "fields": "DescriptionVector"
    }
  ],
  "queryType": "semantic"
}
```

这种混合检索的好处是互补：向量搜索擅长语义匹配（“告诉我关于 AI 安全的内容”），关键词搜索擅长精确匹配（“查找 `ERR_CONNECTION_REFUSED` 的解决方案”），语义排序器（Semantic Ranker）则进一步通过机器阅读理解提升结果质量。

在近年的企业实践中，混合检索栈也正在扩展为 **Sparse（BM25）+ Dense（向量）+ Graph（图遍历）+ SQL（结构化查询）+ Grep（代码搜索）** 的组合，根据查询类型动态选择更合适的路径 [5]。

### 2.4 Agentic Retrieval：Agentic RAG 的核心变化

如果说 Hybrid Search 是在系统层面做“多路召回 + 智能融合”，那么 **Agentic Retrieval** 则是把检索的决策权交给了模型本身（这也是构成广义 Agentic RAG 端到端架构的最核心环节）。

> **通俗理解**：以前的 RAG 是人类考官把找好的参考书塞到大模型手里；而 Agentic 模式，则是让大模型自己去图书馆查资料。遇到问题时，它自己决定先查词典，查不到再去翻年鉴，直至找到最终答案。

为了实现这种“让模型自己搜”，学术界和工业界都走出了从抽象到工程落地的明确步伐。

**学术界的初步实验：提供层次化工具（A-RAG）**

2026 年 2 月，一篇名为 A-RAG 的研究论文 [10] 提出了一个直观的框架：不再为系统写死检索流程，而是直接向大模型暴露三种基础的检索工具——


| 工具                  | 用途         | 适用场景              |
| ------------------- | ---------- | ----------------- |
| **Keyword Search**  | BM25 关键词匹配 | 精确匹配：错误码、ID、专有名词  |
| **Semantic Search** | 向量语义匹配     | 概念性查询：模糊描述、跨语言    |
| **Chunk Read**      | 读取特定文档块    | 细粒度信息提取：具体段落、代码片段 |


模型不是被动接受检索结果，而是**自主决定用哪个工具、搜什么关键词、读哪些文档块**。论文实验表明，A-RAG 在多个开放式 QA benchmark 上，用更少的检索 token 超越了现有方法（benchmark，即基准测试集）。

A-RAG 的关键洞察是：**现有 RAG 系统要么是单次检索拼接到输入中，要么预定义工作流让模型按步执行，两种范式都不允许模型参与检索决策。** 而模型的推理能力恰恰是检索决策中最宝贵的资源。

**建立全局视野：系统化定义 Agentic RAG**

随着这种“模型主导检索”思路的爆发，业界亟需一个统一的框架来收敛各种五花八门的实现。2026 年 3 月，学术界发布了一篇典型的 SoK（Systematization of Knowledge，即“对现有散乱知识进行系统化梳理”）性质的综述论文 [3]。该文首次对 Agentic RAG 这个整体系统进行了严谨的定义，将其形式化为**有限视野的部分可观察马尔可夫决策过程（Finite-horizon POMDP）**——通俗地说，就是用数学模型定义了**“如何让大模型在手里线索不全的情况下，一步一步做出最优的搜索决策”**。它将这一过程拆解为支撑该架构的四大核心模块：

- **Planning（规划）**：模型如何规划检索策略——分解查询、确定检索顺序、评估何时停止
- **Retrieval Orchestration（检索编排）**：如何协调多步检索——并行 vs 串行、工具选择、结果整合
- **Memory（记忆）**：如何管理和利用历史信息——短期工作记忆、长期持久记忆、跨会话记忆
- **Tool Invocation（工具调用）**：如何调用外部工具——API 调用、数据库查询、文件系统访问

这篇综述还指出了 Agentic RAG 的系统性风险：**级联幻觉传播**（一次检索错误导致后续推理全部偏移）、**记忆投毒**（恶意注入的错误信息被模型“记住”）、**检索错位**（模型选择的检索策略与实际需求不匹配）。

**Progressive Disclosure：渐进式信息发现**

Anthropic 在 Claude Code 的实践中提出了另一个重要概念——**Progressive Disclosure** [4]。

Claude Code 最初使用传统 RAG：预索引代码库，每次响应前检索相关片段喂给模型。但这种方式有几个问题：需要索引和配置，在不同环境下容易出故障，更重要的是，Claude 被动接受上下文，而不是主动发现上下文。

Anthropic 的替代方案是给 Claude 一个 **Grep 工具**，让它像开发者一样在代码库中搜索：按文件名、按符号名、按内容模式。这不是简单的“把 grep 包装成 API”，而是让模型拥有了一种全新的检索原语。

更进一步，Anthropic 通过 **Agent Skills** 和 **Subagents** 实现了渐进式信息加载：Claude 可以读取 skill 文件，skill 文件引用其他文件，形成递归的发现链。例如 Claude Code Guide subagent 专门负责文档搜索——在自己的上下文中完成检索，只把最终答案返回给主 agent，保持主上下文干净。

这种设计的哲学是：**不预先决定模型需要什么信息，而是让模型在需要时自己去发现。** 上下文窗口是最宝贵的资源，只在需要时才加载信息，避免 context rot（上下文污染，**指一次性塞给模型太多无关的背景知识，导致其被带偏或注意力分散而产生幻觉**）。

**Azure Agentic Retrieval：产品级落地**

微软在 2026 年推出了 Azure Agentic Retrieval，这说明 Agentic RAG 的核心检索能力，已经开始进入受控的产品级服务阶段 [11]。该管线的工作流程如下：

1. **查询规划**：LLM 分析对话历史，将复杂查询分解为多个子查询
2. **并行执行**：所有子查询同时执行，每个子查询可以是关键词、向量或混合搜索
3. **语义重排序**：每个子查询的结果独立进行语义重排序
4. **结果合成**：合并所有结果，输出统一响应（包含 grounding data、引用来源、执行计划）

这个管线支持 Knowledge Base 对象和 MCP 端点，可以直接被其他 agent 和 chat 应用消费。微软将其定位为 Microsoft Foundry IQ 的知识层，这说明 Agentic RAG 的核心检索思路已经开始进入受控的产品化阶段；但距离成为大多数团队的默认架构选择，仍然还有明显距离。

## 三、应用场景与架构匹配

不同的应用场景对检索能力的需求差异很大。下面按场景分析哪些架构更适合。

### 3.1 知识库与文档问答

**推荐架构：Advanced RAG / Hybrid RAG**

这是 RAG 最经典的应用场景。企业将内部文档（技术文档、产品手册、政策文件等）索引后，员工通过自然语言查询获取准确答案。

这类场景的特点是：文档量中等（通常在百万 token 以内）、查询类型多样（既有精确查询也有模糊查询）、对准确性要求高但对实时性要求一般。Hybrid RAG（向量 + 关键词 + 重排序） 是当前最务实的选择——它覆盖了大部分查询类型，且工程复杂度可控。

在这个场景中，业务团队往往低估了 **切分策略（Chunking Strategy）** 对最终效果的决定性影响。早期机械式的按固定长度切分（如 512 token）经常将一个完整的段落或表格从中间拦腰斩断，导致向量化后的片段毫无语义可言。为了解决 chunk 边界问题，业界发展出了多种切分方案：

- **滑动窗口切分（Sliding Window Chunking）**：在文档上移动一个固定大小的窗口（如 512 个 token），窗口之间设置重叠部分（如 50% 重叠）。这种方法可以确保重要的上下文信息不会因为 chunk 边界而被切断，特别适合长文档和结构化内容。但缺点是会产生更多 chunk，增加存储和检索成本，重叠部分可能导致信息冗余。
- **基于规则的结构感知切分**：根据文档结构（如 PDF 的表格、HTML 的层级、Markdown 的标题）进行智能切分，保持文档的原始结构关系。
- **基于 LLM 的语义切分（Semantic Chunking）**：利用大模型理解文档内容，根据语义边界进行切分，确保每个 chunk 都包含完整的语义单元。

真正的工程难点在于跨模态和结构化元素的解析：如何保持 PDF 中复杂表格的行列对齐关系？如何让包含父子层级的目录结构在切分时不丢失上下文层级信息？在此背景下，基于规则的结构感知切分以及基于 LLM 的语义切分（Semantic Chunking） 逐渐成为标配。

**Dify** [12] 提供了完整的标准化 RAG 管线：文档摄入 → 切分与索引 → 检索 → 重排序 → 生成，其优势是极低的心智负担和可视化调度。而 **RAGFlow** [13] 则切中了上述痛点，它极其专注于深度文档理解（Deep Document Understanding），利用基于 OCR 和模板识别的智能切分技术提取复杂文档结构，并提供可视化 chunking 界面供人工校准，在严肃的企业知识库中显得更具厚度。

### 3.2 代码辅助

**推荐架构：Grep-first Retrieval（基于结构化分析的 Agentic 检索）**

代码检索是 RAG 演进中最具颠覆性的一段探索。早期实践中，开发者试图直接用向量切块（Vector Chunks） 来索引代码库，但很快发现在处理跨文件依赖、符号引用追踪时力不从心。代码不同于自然语言，它是强语法结构化的：函数签名、嵌套的作用域和跨文件的 `import` 关系，在被转化为一堆同质化的浮点数向量后，往往会发生显著的结构信息丢失。

随后，业界达成共识:**在寻找指定名称的类或函数调用时，传统的 Grep 搜索与 AST（抽象语法树） 解析远比向量距离计算更可控、更精确。**

**Grep-first Retrieval** 正在成为代码场景里越来越常见、也更稳的工程选择。最典型的如：Claude Code 不再单纯依赖向量路由，而是向模型提供操作系统级的工具集：让大模型自己运用 Glob 匹配特定目录、直接用 Grep 查找精确的符号名、进而去读取相关文件的上下文。这类路径在符号定位、调用链追踪和精确改动分析上通常比纯语义检索更可靠，同时检索路径也更容易回放和解释 [14]。本质上，它构建了一种强约束的 Agentic RAG——模型掌握了“查什么”的主动权，但在“如何执行查找”的实现载体上，回到了最扎实、最传统的工程手段。

### 3.3 Agent 记忆系统

**推荐架构：状态机管理 + 时序图谱（超越单纯的混合检索）**

Agent 记忆与知识库问答有本质区别：知识库是“检索外部静态事实”，而记忆是“管理自身动态经验”。如果仅拿着做 RAG 的惯性思路（比如将对话片段存入向量库再匹配召回）来做记忆系统，很快就会在实际业务中翻车：当用户周一说“我对花生过敏”，周三又改口说“复查了，其实不过敏只是肠胃炎”，单纯的检索会把这两条互相矛盾的切片同时扔给大模型，造成严重混乱。**在架构层面上，Agent 记忆系统与其说是多路召回，不如说更接近状态管理（State Management）和增删改查（CRUD）。**

**Mem0** [15] 是这一领域的代表项目。它表面上提供混合检索(语义 + BM25 + 实体)，但其真正的核心机制是**意图识别与状态更新**：在沉淀数据前，系统会利用轻量级模型进行意图碰撞检测。如果有状态冲突或变迁，它会对历史记录做软覆盖。它具备 User / Session / Agent 三级结构，非常适合需要持续动态维护用户画像的场景（如偏好追踪、伴随式客服）。论文 [16] 显示其底层设计比早期的单纯向量拼接大幅提升了准确率。

**Zep / Graphiti** [9] 则从另一个维度切入，走向了**时序知识图谱（Temporal Knowledge Graph）路线，本质上是在用图谱跑事件溯源（Event Sourcing，简而言之就是不仅记录当前的最终状态，还像流水账一样记录这件事情从古至今发生的每一个动作）**。面对记忆更迭，它不直接删除旧事实，而是巧妙地通过图谱边（Edges）的有效性窗口（如引入 `valid_at` 和 `invalid_at` 属性）来记录时间的流逝。这对于需要严格追溯演进轨迹的 B 端系统至关重要（例如客户关系管理 CRM 中，系统不仅要回答“当前状态”，还要能复盘“过去每个时间点的状态是如何变更的”）。

选型建议：**对于极简的对话记录或小型内部系统，传统的向量堆叠可能已经够用；但当场景涉及状态的动态覆写时，推荐重点考察 Mem0；对于需要历史回放和长程逻辑推演的复杂业务，Graphiti 这类时序图谱则更具合理性。Agent 记忆系统的选型，本质是对系统状态管理复杂度的评估。**

### 3.4 搜索增强

**推荐架构：Hybrid RAG + Re-ranking**

Perplexity、You.com 等搜索引擎是 RAG 在消费端的应用。这类场景的特点是查询类型极不确定、数据源是 web（而非私有知识库）、对延迟敏感。混合检索（关键词 + 向量）配合重排序是标准做法。

## 五、未来：共存与演化

### 5.1 混合检索是工程层面的务实选择

没有单一检索模式能覆盖所有场景。向量检索擅长语义匹配但不擅长精确匹配，关键词搜索擅长精确匹配但缺乏语义理解，图遍历擅长关系推理但建图成本高，SQL 擅长结构化查询但需要预定义 schema。

未来的检索系统必然是**混合的**——不是简单地把多种检索结果拼在一起，而是根据查询类型动态选择最优的检索路径。但“混合”本身也带来新的工程复杂度。

以 Azure AI Search 为例，其 Hybrid Search 用 RRF（Reciprocal Rank Fusion，倒数排名融合）合并全文搜索和向量搜索的结果 [5]。RRF 的核心思路是对每路检索结果的排名取倒数求和，实现简单且无需训练。但它有明显局限：对不同数据分布的适应性有限——当关键词搜索和向量搜索的结果分布差异较大时（例如一路结果非常稀疏），RRF 的机械式合并可能不如单一检索效果好。此外，RRF 的关键参数(k 值)调节相当经验性，没有普适的最佳值，需要针对具体数据集和查询分布进行实验。在实践中，"混合检索一定比单一检索好"这个假设并不总是成立。

尽管如此，混合检索仍然是当前最值得优先考虑的方向之一。开源生态也在跟进：Mem0 的混合检索（语义 + BM25 + 实体）、Graphiti 的混合检索（语义 + BM25 + 图遍历）都是例证。只是需要理解，**混合检索不是银弹，而是一个需要持续调优的工程选择**。

### 5.2 Agentic RAG：方向越来越清晰，但现实仍然复杂

比混合检索更深刻的变化是：**搜索决策的更多环节，开始从系统工程师手里转移到模型侧。** 在 Agentic RAG 架构里，由模型主导或深度参与的搜索动作，可以视为其最核心的组成环节之一，我们称之为 Agentic Retrieval（代理式检索）。

A-RAG 的三层检索接口 [10]、Claude Code 的 Progressive Disclosure [4]、Azure 的 Agentic Retrieval [11] 都指向同一个方向：模型不再只是被动的“检索结果消费者”，而开始参与“检索策略制定者”的角色。

这种变化的吸引力是显而易见的：

- **适应性**：模型可以根据具体任务动态调整检索策略，而不是被限制在预定义的管线中
- **可扩展性**：随着模型能力提升，检索质量自然提升，无需重新设计系统
- **简洁性**：不需要为每种查询类型设计专门的检索管线，模型自己会选

但冷静来看，“Agentic RAG 是范式革命”这个判断可能过于乐观了。原因有三：

**第一，模型能力仍然是瓶颈。** Agentic RAG 的效果强依赖于模型的 function calling（函数调用，即模型按预设格式输出结构化指令以触发外部工具）稳定性、工具选择判断力和幻觉率。当前模型在这些维度上远未完美——模型可能选错工具、构造出无效的搜索查询、或者在检索到不相关内容后仍然"自信地"继续推理。把检索决策权交给模型，本质上是把不确定性的来源从"可调试的系统设计"转移到了"不透明的模型黑盒"。系统工程师至少还能查看检索日志、调整 BM25 参数、优化 chunk 策略；但模型“为什么决定用 keyword search 而不是 semantic search”这个问题，往往难以追溯和干预。

**第二，工程落地面临实际困难。** 模型自主检索意味着不可预测的工具调用次数和 token 消耗。一个复杂的查询可能触发 3 次检索，也可能触发 30 次——这直接影响延迟和成本。SoK 综述 [3] 指出的"级联幻觉传播"问题在实践中尤为棘手：如果第一次检索就召回了一条看似相关但实际错误的信息，模型可能会基于这个错误信息进行后续推理，产生一连串的错误推导，而最终输出的"幻觉"可能看起来非常合理。更危险的是"记忆投毒"场景——如果恶意数据被注入知识库，模型在自主检索中可能反复召回并强化这些错误信息，甚至将其“记住”到长期记忆中（详见 5.4 节）。

**第三，可观测性和可控性严重不足。** 当模型的检索路径不可预测时，debug 变成了噩梦。传统 RAG 管线中，你可以清晰地看到：查询被改写成了什么、召回了哪些文档、重排序后的 top-5 是什么。但 Agentic RAG 中，检索路径是模型在运行时动态决定的，你需要完整的工具调用链追踪才能还原整个过程。目前业界的可观测性工具（如 LangSmith）在 Agentic 场景下的支持也还在早期阶段。

### 5.3 长上下文 vs RAG：不是替代，但边界在变化

随着 128K、200K 甚至更长上下文窗口的普及，“长上下文是否替代 RAG”成为了一个热门话题。简单回答“不会”是偷懒的，真实情况更微妙。

**百万 token 级的知识库，长上下文确实替代不了 RAG。** 把大量文档全部塞进上下文窗口既不经济（token 成本线性增长），也不高效（“大海捞针”问题——当上下文中信息密度很低时，模型的有效推理能力会下降)。RAG 的核心价值——**精准检索**——用最小的 token 开销找到最相关的信息——在这个量级上不可替代。

**但几百 K token 的场景，长上下文正在成为 RAG 的合理替代。** Gemini 2.5 Pro 提供了 100 万 token 的上下文窗口，Claude 和 GPT 系列也支持 200K。对于一个中小型项目的技术文档（几十到几百 K token），直接把文档全放进去，比搭一套 RAG 管线要简单得多，效果也未必差。这种方式省去了 chunk 策略调优、embedding 模型选择、向量数据库运维等一系列工程复杂度，在早期探索和低成本场景下是一个务实的选择。

**更值得关注的是混合方案**:先用长上下文处理核心文档集，再用 RAG 按需检索补充信息。这种方案兼得两者的优势，也正在被一些产品采纳。

所以更准确的判断是：**长上下文没有替代 RAG，但压缩了 RAG 的必要场景。** 当知识库规模在长上下文窗口覆盖范围内时，RAG 不再是知识增强的唯一路径，而更像一种应对特定成本、延迟和可更新性约束的优化选项。

#### 一个最小决策框架

如果团队需要在“直接塞长上下文”和“建设检索系统”之间快速做判断，至少应看四个约束：

- **文档规模**：核心知识是否能稳定落在上下文窗口内，而不是靠极限堆 token 勉强塞下。
- **更新频率**：文档是否高频变更；如果每天都在改，长上下文人工维护会迅速失控。
- **交互延迟**：业务是否对 TTFT 极度敏感；如果是对话式产品，首字延迟往往比绝对准确率更先暴露问题。
- **单次成本上限**：系统能否承受把几十万 token 反复送入模型；如果不能，检索就不是“可选优化”，而是成本控制手段。

满足前两项且对后两项不敏感时，长上下文可以先行；只要更新频率、延迟或成本中的任一约束变得尖锐，RAG 或混合检索通常就会重新变得必要。

### 5.4 被忽视的地基：数据治理

通篇讨论“怎么搜”，但有一个问题几乎没有被触及：**搜的东西本身靠谱吗？**

传统 RAG 中，知识库质量差会导致召回错误结果，但影响范围是一次性的——这次答错了，下次换个查询可能就对了。但在 Agentic RAG 中，知识库质量问题会被**多步推理链放大**：模型基于低质量检索结果做出中间判断，再基于这个错误判断继续检索和推理，错误层层叠加，最终产出一个看似合理但完全错误的答案。“垃圾进垃圾出”这个老问题，在 Agentic RAG 中变得比以往任何时候都严重。

更具体地说，数据治理在 RAG 系统中至少包含三个层面：

- **数据质量**：文档本身的准确性、时效性、完整性。过时的技术文档、矛盾的内部政策、未标注来源的二手信息——这些在传统知识库中就存在，但在 RAG 系统中因为“有 LLM 背书”而更容易被用户信任，也更危险。
- **切分与元数据**：chunk 策略的选择、元数据的标注（如文档版本、更新时间、来源分类）直接影响检索质量（切分策略的具体工程挑战在 3.1 节中已展开）。一个没有标注版本的 API 文档 chunk，可能召回的是已废弃接口的描述。
- **访问控制与安全**：谁可以向知识库中添加内容？如何防止恶意数据注入？在 Agentic RAG 中，“记忆投毒”风险（通过对话注入虚假信息，让模型将其存入记忆并在后续检索中反复召回）比传统 RAG 中更加现实。

这些问题不是 RAG 架构本身能解决的，但如果不解决，再先进的检索架构也只是在更高效地放大噪音。**无论是混合检索还是 Agentic RAG，其上限都建立在数据质量、元数据治理和权限控制先过关的前提上。** 在讨论检索范式升级之前，先确认知识库本身的质量基线，是更务实的优先级。

### 5.5 落地现实：技术范式与工程约束的平衡

在探讨 Agentic RAG、混合检索与长上下文等前沿范式时，单纯的技术理想往往会遭遇工业界的真实骨感。任何检索系统的演进，最终都需要在模型表现、基础设施建设以及用户体验指标之间寻找平衡点。

#### 基础设施的不可见负载

引导模型像开发者一样主动调用工具发现线索（如 Claude Code [4]），展示了极具启发性的潜力。但工程团队在引入这种范式时应当审慎。根据相关技术与工业界架构的复盘 [14]，能够支撑模型高效、受控搜索的背后，往往依赖极度重型的传统分析引擎（例如利用 LSIF（Language Server Index Format，语言服务器索引格式）预解析代码、白名单过滤及依赖图打平）。若缺乏底层基建对无效日志和无用产物进行强力“降噪”，直接将原始操作系统接口暴露给模型，不仅会导致严重的上下文污染，还会带来失控的 API 成本。Agentic 检索并非模型能力的单方面突破，而是智能调度与传统重型基建的深度绑定。

#### 企业级系统的细粒度权限要求

当试图将成熟的检索框架部署进企业内网时，“数据隔离与权限控制”往往比“检索准确率”具有更高的一票否决权。现实企业环境中的知识库，通常被 AD/LDAP 等身份系统赋予了极为动态的访问控制（RBAC，基于角色的访问控制；或 DLS，文档级安全策略）。根据工业部署经验，一个安全合规的系统必须保证检索节点能够精确识别并双向同步权限树，从而主动剔除查询者无权触达的内容。目前许多追求敏捷开发的开源框架在工作流上表现优异，但在同深层企业权限架构实时对话的能力上依然存在短板，这限制了它们在核心业务网中的渗透。

#### TTFT 延迟与交互边界

“无论多长的文档，直接放入大模型进行 In-context（上下文内，即把文档直接放进模型的输入窗口）问答”是架构设计上最具吸引力的极简路线。

然而，如果跳出单纯的评测榜单走向真实商业产品，首字延迟（TTFT，Time To First Token，即用户发出提问后等待模型输出第一个字的空白时间）常常成为阻碍长上下文直接落地的最大路障。根据工业部署经验，当且仅当命中 KV Cache（键值缓存，即模型对已处理 token 的中间计算结果缓存）池时，超长输入的交互才是平滑的；否则单次清洗解析几十万 token 会导致数秒甚至更长时间的阻力延迟。在高度要求连贯反馈的对话侧产品中，此类延迟会直接造成用户流失。

由此可见，即便在长上下文普及的时代，RAG 的价值依然不可磨灭——它从“克服模型容量黑洞”演变成了一种可靠的降低请求端算力负载、压缩 TTFT 的物理保障措施。

### 5.6 一个值得观察的方向：检索与推理的深度融合

回顾 RAG 的演进历程，检索与生成之间的关系在不断拉近：

- **Naive RAG**：检索是一个预处理步骤，与生成是分离的
- **Advanced RAG**：检索和生成之间的交互加深了（re-ranking、multi-hop）
- **Agentic RAG**：检索本身就是推理的一部分——模型在检索中思考，在思考中检索

这个趋势是否会收敛到“检索即推理（Retrieval as Reasoning）”——搜索与推理深度融合为连续过程，模型先快速浏览（keyword search），发现线索后深入阅读（semantic search），找到关键段落时精读（chunk read），必要时回头查找更多信息（multi-hop）？

目前已有初步验证。A-RAG [10] 在 benchmark 上展示了模型自主选择检索策略的有效性，Claude Code [4][14] 在代码场景中证明了 Grep-first 检索的实用性，Azure Agentic Retrieval [11] 则在企业级产品中落地了查询分解 + 并行检索的管线。

但需要诚实地说，这些证据指向的是“模型参与检索决策是有效的”，距离“检索与推理深度融合成为更普遍的工程范式”还有一段距离。A-RAG 是受控 benchmark 实验，Claude Code 是代码场景的特定实践，Azure 是商业产品的官方说法。**这个方向是否会变得更主流，很大程度上取决于模型能力的成熟速度——特别是 function calling 的稳定性、幻觉率的降低，以及成本效率的持续改善。** 在这些条件成熟之前，混合检索 + 人工调优的管线仍然是大多数生产环境的最优选择。

## 总结

RAG 六年的演进史，可以用一个表格来概括不同的架构选择：


| 阶段           | 谁决定搜什么            | 谁决定怎么搜           | 检索模式         | 适合的场景                           |
| ------------ | ----------------- | ---------------- | ------------ | ------------------------------- |
| Naive RAG    | 系统 (embedding)    | 系统 (top-k)       | 纯向量          | 简单问答、快速原型                       |
| Advanced RAG | 系统 (+ re-ranking) | 系统 (+ multi-hop) | 向量为主         | 企业知识库 (通用)                      |
| Graph RAG    | 系统 (实体抽取)         | 系统 (图遍历)         | 向量 + 图       | 关系密集型问答 (预算充足)                  |
| Hybrid RAG   | 系统 (多路召回)         | 系统 (RRF 融合)      | 向量 + 关键词 + 图 | 对检索质量要求高的通用场景                   |
| Agentic RAG  | **模型**            | **模型**           | **模型自主选择**   | 模型 function calling 稳定、幻觉率可控的场景 |


需要强调的是：**这五种范式是共存的，不是替代关系。** 很多企业今天依然在用 Naive RAG 或 Advanced RAG，不是因为他们落后，而是因为够用。选择哪种架构，取决于数据特征、查询类型、预算约束和模型能力——而不是简单地追求"最新"。

从系统预定义到模型自主决策，这是 RAG 发展的一个重要方向，但“方向”不等于“终点”。A-RAG [10]、Claude Code [4][14]、Azure Agentic Retrieval [11] 都展示了模型参与检索决策的有效性；但“模型能参与搜索”与“模型能稳定地主导生产级搜索”之间，仍然隔着模型能力、成本控制、权限边界、可观测性和灾备设计等一整套工程条件。在这些条件成熟之前，混合检索 + 人工调优的管线，仍然是大多数生产环境里更稳的选择。

正如 Anthropic 的 Thariq Shihipar 所说：“Designing the tools for your models is as much an art as it is a science. It depends heavily on the model you're using, the goal of the agent and the environment it's operating in.” [4]

RAG 的演进不是从系统走向模型的单一路径，而是在不同场景下找到系统与模型之间更合适的分工点。最好的检索系统，不一定是概念上最前沿的系统，而是让模型“愿意用、用得好”，同时也让人类“看得懂、能审计、可干预”的系统。

### 其他建议

1. **不要盲目跟风“全接管”**：如果你的业务场景只需要查询标准的红头文件或结构化知识，老老实实用 Naive RAG + 基于规则的过滤。它稳定、便宜且好排错。没有必要为了追逐 Agentic RAG 的概念去引入不确定的模型黑盒。
2. **摸清你的数据底牌**：试图让大模型通过自主调用接口来发现知识之前，先盘点一下内部的基础设施——API 边界是否足够干净？文档数据是否做了严格的“防投毒”和权限隔离？重型数据基建的承载力，比大模型的智力更决定成败。
3. **拥抱“混合组装”心态**：长上下文能力、经典向量检索、工程化 Grep 以及 Agentic 自由搜索在很长一段时间内**都不互斥**。先用长窗口模型吃透核心上下文，再通过精准的 Agentic 调用补充碎片线索，是当下落地阻力最小、容错率最高的工业级组合。
